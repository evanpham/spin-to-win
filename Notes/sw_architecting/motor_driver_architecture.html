Motor Driver — SW Architecture

ESP32 / FreeRTOS · 4 half-bridge · 48V / 40A · Revision 0.5

SWC vs Task — Distinction

Key distinction: A software component (SWC) is an abstract block — inputs, compute, published outputs — with no scheduling identity. A task is a FreeRTOS scheduling entity with its own stack and priority. Two SWCs that are rate-locked and must execute in a fixed sequence belong on the same task. Splitting them adds a context switch, IPC overhead, and phase delay for no benefit. Task boundaries below are drawn only where rate, priority, or core assignment genuinely differ.

Task Map

⬤ Core 1 — Realtime
Task: control_loop PRIO 5
Realtime  MCPWM carrier valley drives CONVST · CPU polling SPI · ~20 kHz · see ADC timing document
Timing is driven by the MCPWM peripheral: the carrier valley event asserts CONVST on the AD7617 via hardware output compare with no CPU involvement. The loop then polls BUSY until conversion completes (~7.5 µs), executes a CPU polling SPI transaction to read all 16 channel results (~14.4 µs worst case, 1-wire), and runs FOC math on the fresh data. foc_u_guy reads fault_state_t at the top of each cycle and zeroes outputs if a fault is active. Setpoints read directly from setpoint_t.
  • adc_manager — assert CONVST (via MCPWM output compare, hardware), poll BUSY, execute CPU polling SPI transaction (32 bytes, 40 MHz), write adc_frame_t
  • current_monitor — scale I_abc, filter, set fault flags
  • voltage_monitor — scale V_phase, V_dc, filter
  • rotor_observer — propagate → predict → innovate → correct; outputs θ_est, ω_est
  • foc_u_guy — check fault_state_t, read setpoint_t, Clarke, Park, PI(Id,Iq) → Vd, Vq
  • svm_orchestrator — inverse Park (dq→αβ), SVM sector + dwell → bridge_cmd_t[A,B,C]
  • bridge_driver ×N — check gate_kill, insert dead time, write PWM peripheral
⬤ Core 0 — Background
Task: fault_responder PRIO 3
Safety  event-driven · blocks on task notification
Blocks on xTaskNotifyWait. Woken when any monitor sets a bit in fault_flags_t and calls xTaskNotify with eSetBits. Sole writer to fault_state_t. Executes fault response (see open questions — fault taxonomy). Primary gate kill mechanism is the LTC7066 EN GPIO (hardware-speed, independent of Core 1 firmware). Software gate_kill flag read by bridge_driver is a belt-and-suspenders for graceful shutdown only.
  • fault_responder — read fault_flags_t, write fault_state_t, drive safing
Task: control_loop_watchdog PRIO 3
Safety  Core 0 · timestamp polling · independent of Core 1
Runs on Core 0 so it is unaffected by a Core 1 hang. Each cycle of control_loop writes last_frame_time (a tick count). Watchdog wakes periodically, checks now − last_frame_time against a threshold (~10× expected frame period). On timeout: pulls LTC7066 EN GPIO low directly (hardware gate kill, independent of Core 1), then notifies fault_responder with FAULT_CONTROL_LOOP_TIMEOUT. Timestamp approach preferred over active notification from control_loop — watchdog fires even if control_loop hangs before it can write the timestamp. See ADC timing document for threshold sizing.
  • control_loop_watchdog — poll last_frame_time, timeout → gate kill + fault notify
Task: telemetry PRIO 2
Core 0  rate-limited · ESP-NOW / UART
Snapshots motor_state_t and monitor outputs (non-blocking). Serialises and transmits. Receives external commands; writes setpoint_t under mutex.
  • comms_tx — serialise and send telemetry
  • comms_rx — receive, validate, write setpoint_t
Task: temp_monitor PRIO 1
Monitor  low-rate · NTC or I²C
Reads thermistor / digital temp sensor (~1–10 Hz). Sets FAULT_OVERTEMP in fault_flags_t. Updates thermal headroom for current derate (future).
  • temp_monitor — read sensor, threshold compare, flag write
Priority 5 — hard realtime
Priority 3 — safety response
Priority 2 — comms
Priority 1 — background

Signal Flow — Realtime Path

Execution order within control_loop task (each ADC frame): adc_manager → current_monitor → voltage_monitor → rotor_observer → foc_u_guy → svm_orchestrator → bridge_driver. All SWCs share the task stack; no IPC or context switch between them. fault_responder and control_loop_watchdog run on Core 0, independent of Core 1. foc_u_guy reads fault_state_t at top of each cycle and zeroes outputs if a fault is active.

Shared Data Structures & Interfaces

Struct
Producer → Consumer
Mechanism & Notes
adc_frame_t
16 ch results, raw counts
adc_manager → monitors, foc_u_guy (same task)
Written by adc_manager via CPU polling SPI transaction after BUSY falls. Single buffer — no double buffer needed since the control loop is sequential and never reads a frame while another is being written. All downstream SWCs read from this buffer within the same task cycle.
rotor_state_t
θ_est, ω_est, observer_valid
rotor_observer → foc_u_guy (same task)
Plain struct write/read, same task. observer_valid read by foc_u_guy — if false, foc_u_guy runs open-loop V/f until convergence. See rotor_observer document.
svm_cmd_t
Vd, Vq, θ_est
foc_u_guy → svm_orchestrator (same task)
Plain struct, same task. Vd/Vq are direct PI outputs in dq frame. svm_orchestrator owns all transforms: inverse Park (dq→αβ, internal) then SVM (αβ→duties). αβ frame does not cross any SWC boundary.
bridge_cmd_t
duty[A,B,C], enable, brake
svm_orchestrator → bridge_driver ×N (same task)
Shadow register + atomic commit. All phases written before any bridge_driver reads. See bridge driver document for LEDC/RMT atomic commit TODO.
fault_flags_t
uint32 bitmask
monitors (control_loop) → fault_responder (Core 0)
atomic_fetch_or to set. Monitors call xTaskNotify(eSetBits) to wake fault_responder. Cleared only by fault_responder after latching response. xTaskNotify preferred over xEventGroup — single consumer, lighter, 32-bit mask.
fault_state_t
enum + context
fault_responder → foc_u_guy, telemetry
fault_responder is sole writer. foc_u_guy reads at top of each cycle — zeroes outputs and skips loop if fault active. telemetry reads for logging.
last_frame_time
uint32 tick count
control_loop (Core 1) → control_loop_watchdog (Core 0)
Written by foc_u_guy each cycle. Watchdog polls periodically and checks now − last_frame_time. Atomic uint32 write/read sufficient. See ADC timing document for threshold sizing.
motor_state_t
θ_est, ω_est, Iq, Id, Vdc
foc_u_guy → telemetry, watchdog (cross-task)
Written each cycle by foc_u_guy. Telemetry snapshots non-blocking. Consider seqlock if struct grows large.
setpoint_t
Iq_ref, Id_ref, mode
comms_rx → foc_u_guy (cross-task)
Mutex-guarded. Written by telemetry task on command receipt. Read directly by foc_u_guy each cycle. No intermediate state machine layer.

Open Questions & Deferred Decisions

ISR vs task for control_loop execution Running FOC math directly in the DMA completion ISR eliminates the ~5 µs task-notify overhead (~12.5% of the 40 µs frame budget). ISR context restricts some FreeRTOS primitives and requires manual FPU register management, but these are configurable on ESP32 and may be acceptable if Core 1 is dedicated. Counterarguments: ISR stack is a global fixed config, interrupt nesting behavior differs, hardware watchdog (TWDT) integration is cleaner in task context. Not yet resolved — decision should be made empirically once cycle budget can be measured on real hardware. Architecture is compatible with either approach; only the scheduling wrapper changes.
Fault taxonomy and response actions Enumerate fault types and match each to an appropriate hardware response. Candidate responses: (1) open all gates — motor freewheels, no braking, safest for most faults; (2) pull all low-side gates on — active braking / winding short, high current risk; (3) ramp down setpoint — controlled deceleration before gate disable; (4) log only — non-critical, no hardware action. Fault types to classify: overcurrent, overvoltage, undervoltage, overtemp, observer divergence, SPI fault, ADC timeout / control loop hang, shoot-through detection. To be worked through as a dedicated design session.
Observer for sensorless FOC See rotor_observer document for full discussion. SMO is the practical first implementation (VESC precedent, low compute). EKF deferred. Luenberger as bring-up stepping stone. Variant selection deferred until phase current readings are available from ADC test board.
ADC timing, loop scheduling, and watchdog AD7617 SAR ADC selected. MCPWM peripheral drives CONVST at carrier valley via hardware output compare. CPU polling SPI reads 32 bytes after BUSY falls. Worst-case CONVST-to-data-ready: ~21.9 µs (1-wire mode), leaving ~28 µs for FOC math in a 50 µs budget. 2-wire mode fallback saves ~3.2 µs if needed. Full analysis, timing budget, watchdog design, and open questions in ADC timing & watchdog document.
Per-channel scan rate with 2 × AD4115 Confirmed 24,845 SPS per channel pipelined. Does enabling all 18 channels degrade this? Verify from datasheet timing model. Determines whether all channels are available to the current loop at full rate.
LEDC vs RMT for bridge_driver LEDC sufficient for BLDC/PMSM/brushed DC. RMT needed for switched reluctance and stepper. See bridge driver document. Deferred to gate driver test board.
load_configurator — deferred, not in realtime path A configuration-time component (not a realtime SWC) that sets up the control loop for a given motor topology at startup or on explicit reconfigure command. Would instantiate or swap SWCs for different load types: 3-phase PMSM/BLDC (FOC, current architecture), stepper (two-winding orchestrator), brushed DC (direct duty), switched reluctance (sr_sequencer). BLDC and PMSM grouped under the same FOC architecture — both are permanent magnet machines, the dq-frame current loop is identical, differences are observer tuning and field weakening strategy only. Not scoped until basic FOC is running.
RESOLVED — foc_u_buddy removed foc_u_buddy eliminated. Its responsibilities are redistributed: foc_u_guy reads fault_state_t directly and zeroes outputs on fault; comms_rx writes setpoint_t directly for foc_u_guy to consume; topology/mode configuration deferred to load_configurator (non-realtime). No orchestration layer in the realtime loop.
