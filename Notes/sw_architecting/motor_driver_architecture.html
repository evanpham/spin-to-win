Motor Driver — SW Architecture Sketch

ESP32 / FreeRTOS · 4 half-bridge · 48V / 40A · Revision 0.1 (concept)

Dual-Core Task Map

⬤ Core 1 — Realtime
ADC_Manager PRIO 5
Realtime  1 task · both AD4115
Owns SPI bus. Sequences CS lines for both devices. Triggers conversions, DMA-reads results into double-buffered adc_frame_t. Signals FOC_Control on each complete frame. Tracks SPI fault counters.
foc_u_guy PRIO 5
Current Loop  Runs every ADC frame
Wakes on ADC frame notification. Reads adc_frame_t. Calls monitors (update/publish inline). Runs Clarke/Park transforms, PI controllers, inverse Park/SVM. Writes bridge_cmd_t to each BridgeDriver queue. Reads motor_mode_t from foc_u_buddy to gate loop execution.
foc_u_buddy PRIO 4
State Machine  Shared task with foc_u_guy
Motor controller state machine: idle → align → run → fault → coast. Runs as a component within the same task as foc_u_guy, called after the current loop completes each cycle. Reads fault_state_t and motor_state_t to drive transitions. Writes motor_mode_t (consumed by foc_u_guy) and setpoint_t when executing ramp profiles. Owns calibration sequences and field-weakening mode transitions.
svm_orchestrator + bridge_driver ×N PRIO 4
Realtime  shared task · sequential execution
svm_orchestrator receives svm_cmd_t (V_alpha, V_beta) from foc_u_guy, computes SVM sector and dwell times, writes all three bridge_cmd_t structs atomically. Each bridge_driver instance owns one half-bridge: applies dead time, checks gate_kill, writes PWM peripheral. bridge_driver is topology-agnostic and reused across motor types; svm_orchestrator is bypassed for brushed DC and replaced for stepper/SR. See Bridge Driver & SVM Orchestrator document for full detail.
⬤ Core 0 — Background
Fault_Responder PRIO 3
Safety  Sole writer to fault register
Polls/pends on fault_flags_t (set atomically by monitors). Owns safing logic: gate disable, ramp-down command, indicator LED. Publishes fault_state_t for FOC state machine to read. Does not act on hardware directly except gate kill line.
Telemetry / Comms PRIO 2
Core 0  ESP-NOW / UART
Reads snapshots of monitor structs (non-blocking). Serializes and transmits. Receives setpoint commands; writes to setpoint_t shared struct (guarded by mutex).
Temp_Monitor PRIO 1
Monitor  Low-rate, NTC or I²C
Reads thermistor / digital temp sensor. Sets FAULT_OVERTEMP flag if threshold exceeded. Updates thermal headroom estimate for FOC current limit derate (future).

Design intent: foc_u_guy (current loop) and foc_u_buddy (state machine) are distinct software components sharing a single FreeRTOS task. foc_u_guy runs first each cycle on ADC frame notification; foc_u_buddy runs after, reading loop outputs to drive state transitions. Monitors are functions called inline within foc_u_guy — they set bits in fault_flags_t atomically but take no hardware action. Fault_Responder on Core 0 handles safing asynchronously.

Written each FOC cycle. Telemetry takes non-blocking snapshot. No mutex needed if reads are atomic per-field or struct is small enough for single-copy.

Open Questions & Deferred Decisions

ADC frame timing vs PWM sync How do we align ADC conversion trigger with PWM switching events (sample at valley of carrier)? Likely needs a GPIO trigger from LEDC/RMT timer output, or a timer ISR that kicks ADC_Manager. To be resolved once test board is running.
Per-channel scan rate with 2 × AD4115 Confirmed 24,845 SPS per channel in pipelined scan mode, but does enabling all 18 channels degrade this? Need to verify from datasheet timing model. Determines whether all channels can be sampled at 18.5 kHz or if channel subset must be used for the current loop.
FOC_Control: inline monitors vs separate tasks Proposed to call Current/Voltage monitors as functions within FOC_Control. This is efficient but means monitor logic runs at FOC rate. If monitor logic becomes expensive (e.g. RMS integration, harmonic analysis) it may need its own lower-priority task with snapshot reads.
Observer for sensorless FOC Sliding mode, Luenberger, EKF. EKF has best low-speed performance but highest compute cost. To be evaluated once phase current readings are available from test board.
BridgeDriver: LEDC vs RMT LEDC is simpler and sufficient for 3-phase PWM. RMT gives more flexible waveform control (useful for switched reluctance or stepper). Decision deferred to gate driver test board phase.
Motor state machine ownership — RESOLVED Split into two components on one task: foc_u_guy owns the current loop, foc_u_buddy owns the state machine. Both run sequentially within the same FreeRTOS task each ADC frame cycle. foc_u_buddy reads loop outputs and fault state to drive transitions; writes motor_mode_t to gate foc_u_guy execution.
