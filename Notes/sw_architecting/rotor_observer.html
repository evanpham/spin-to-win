rotor_observer — Concepts & SWC Sketch

Observer theory, design decisions, and software component interface · Revision 0.1

Why an Observer — Not Direct Calculation

Rotor position cannot be read directly in a sensorless system. It must be inferred from phase currents and voltages via the motor's electrical model. The naive approach — compute back-EMF from V - IR - L·dI/dt, estimate ω from back-EMF via Kv, integrate ω to get θ — works in principle but has two fundamental problems.

Integration drift. Any error in R, L, or flux linkage produces a persistent error in the back-EMF estimate. Integrating that error accumulates unboundedly into the θ estimate. There is no correction mechanism — the estimate wanders from truth over time.

Low-speed collapse. Back-EMF scales with ω. At low speed it is tiny relative to the resistive and inductive voltage drops, making it effectively buried in noise. Direct back-EMF extraction becomes unreliable below ~10–15% of rated speed for most motors.

An observer solves both problems by adding a correction loop: it continuously compares what the motor model predicts the measurements should be against what was actually measured, and uses the residual to pull the state estimate back toward truth every cycle.

Observer Cycle — Step by Step

Each ADC frame triggers one full observer cycle. All four steps execute within the same task cycle as foc_u_guy.

1
Propagate — advance state using physics
Use last cycle's estimates (θ, ω) plus the present voltage input and the motor model (R, L, flux linkage) to predict what the state should be now, without looking at measurements.
θ_pred = θ_est[n-1] + ω_est[n-1] · dt ω_pred = ω_est[n-1] + torque_model · dt
2
Predict measurement — what should we be seeing?
Run the motor model forward from the propagated state to compute what the phase currents should look like this cycle. This uses the same model inputs as step 1 — the key is that it uses the previous state estimate, not the current measurement.
I_predicted[n] = f(θ_pred, ω_pred, V_phase[n], R, L)
Steps 1 and 2 are operationally one function call. Separated here for conceptual clarity only.
3
Innovation — measurement residual
Compare predicted current to actually measured current. This residual is nonzero whenever the state estimate is wrong — not just when measurements are changing. It is expressed in measurement space (amps), not state space (radians).
innovation[n] = I_measured[n] - I_predicted[n]
4
Correct — update state estimate
Add a scaled fraction of the innovation to the propagated state. L_gain translates from measurement space (amps) to state space (radians, rad/s). It sets the trust balance between model propagation and measurement correction.
θ_est[n] = θ_pred + L_gain_θ · innovation[n] ω_est[n] = ω_pred + L_gain_ω · innovation[n]
High L_gain: snaps to measurement — responsive but noisy. Low L_gain: trusts the model — smooth but drifts if model is wrong. Gain tuning is the primary design parameter of the observer.
Key insight: The observer does not do f⁻¹(f(measurements)) = measurements. The prediction uses the previous state estimate propagated forward; the measurement is current. Any estimation error from the previous cycle shows up as a nonzero innovation — a persistent, self-correcting signal that keeps the estimate anchored to reality.

Model Error and Parameter Bias

If R or L are wrong, I_predicted is systematically off even when the state estimate is correct. The observer interprets this as a state error and biases θ_est and ω_est in whatever direction minimises it — which is the wrong direction. The estimate converges to a wrong value confidently.

Parameter ID at startup
Method
Inject test signal (DC pulse for R, AC for L), fit measured response
Pros
Simple, runs once, no runtime cost. Good for stable parameters.
Cons
R changes with temperature. L changes with saturation. Parameters drift after identification.
Extended state estimation
Method
Add R (or flux linkage) as an additional state variable. EKF estimates [θ, ω, R] simultaneously.
Pros
Self-correcting over time. Handles thermal drift. No separate ID step needed.
Cons
Significantly higher compute cost. EKF already expensive; augmented state compounds it.
Disturbance / integral term
Method
Add integrating disturbance state that accumulates persistent innovation. Absorbs slow biases without identifying their source.
Pros
Low complexity. Effective for slow-moving parameter drift. Common in practical FOC implementations.
Cons
Integrator can wind up. Doesn't distinguish between model error and real disturbances.
Practical note: Most BLDC sensorless implementations tolerate 10–20% parameter error because the correction loop partially compensates. Start with measured R and L from bench identification. Exact parameters matter most at low speed where back-EMF is small and the model term dominates. Do not optimise parameter estimation before basic observer convergence is demonstrated.

Observer Variants — Tradeoffs

Luenberger Observer
L_gain
Fixed matrix, designed by pole placement. Constant across operating conditions.
Compute cost
Low — a few multiplies per cycle. Suitable for ESP32 at full update rate.
Limitations
Gain is fixed — not optimal at all operating points. Poor low-speed performance.
Sliding Mode Observer (SMO)
L_gain
Replaced by a switching term — gain is effectively infinite when crossing the sliding surface.
Compute cost
Low-medium. More robust to model error than Luenberger.
Limitations
Chattering at the sliding surface introduces high-frequency noise into θ_est. Requires filtering. VESC uses SMO as default.
Extended Kalman Filter (EKF)
L_gain
Computed each cycle from covariance matrices — statistically optimal given noise model.
Compute cost
High — matrix operations each cycle. Feasible on ESP32 but eats into cycle budget.
Limitations
Requires tuning of process and measurement noise covariances (Q, R matrices). Best low-speed performance of the three.

rotor_observer — SWC Interface Sketch

rotor_observer component · inline with foc_u_guy
← Inputs (each cycle)
→I_abcfloat[3] — measured phase currents (A)
→V_abcfloat[3] — applied phase voltages (V)
→motor_paramsconst struct — R, L, flux_linkage, pole_pairs
→dtfloat — cycle period (s)
→ Outputs (each cycle)
←theta_estfloat — electrical angle (rad)
←omega_estfloat — electrical angular velocity (rad/s)
←observer_validbool — convergence flag
←innovation_magfloat — residual magnitude (diagnostic)
Internal execution sequence (each call)
Propagate: advance θ_est, ω_est using motor model and V_abc input
Predict: compute I_predicted from propagated state + V_abc + motor_params
Innovate: compute innovation = I_abc - I_predicted
Correct: apply L_gain (or SMO switching term) to update θ_est, ω_est
Validate: set observer_valid false if innovation_mag exceeds threshold (divergence detection)
Persistent state (lives between cycles)
θ_estcurrent angle estimate — updated each cycle
ω_estcurrent velocity estimate — updated each cycle
L_gainobserver gain — fixed (Luenberger/SMO) or computed (EKF)
variantenum: LUENBERGER | SMO | EKF — selects internal algorithm
Placement: rotor_observer is called as a function within foc_u_guy, after monitors and before the Park transform. Its outputs (theta_est, omega_est) feed directly into the Park transform rotation matrix. observer_valid gates foc_u_buddy's state machine — if the observer hasn't converged, foc_u_buddy holds in align state and foc_u_guy runs open-loop V/f until convergence.
