ADC Selection — AD7617

The original design used the AD4115 delta-sigma ADC. That device delivers 24,845 SPS total, distributed across all enabled channels in a pipelined scan — not 24,845 SPS per channel. With 7 minimum required channels (3× phase current, 3× phase voltage, DC bus voltage), the per-channel rate drops to ~3.5 kHz, which is insufficient for controlling any reasonably high-speed motor.

The selected replacement is the AD7617: a 14-bit dual SAR ADC with 16 channels organised as two simultaneous-sampling banks of 8 (bank A: V0A–V7A, bank B: V0B–V7B). A single CONVST pulse simultaneously samples one channel from each bank, giving two results per conversion cycle. At 1 MSPS per channel pair and 8 pairs covering all 16 channels, the effective per-channel sample rate is 125 kHz — well in excess of requirements.

The AD7617 is explicitly listed in its datasheet for multiphase motor control applications. Key tradeoffs versus the AD4115: 14-bit resolution versus 24-bit, higher noise floor, no integrated delta-sigma filtering. Whether the noise floor is acceptable for phase current sensing must be validated on the test board against actual switching noise levels before the design is committed.

Noise floor note: The AD7617 specifies 85 dB SNR at 1 MSPS on the ±10V range. At the ±10V range, 1 LSB = 1.22 mV. The INA241 current sense amplifier gain and shunt resistor selection must be chosen such that the full-scale current maps to a voltage swing that keeps quantization error acceptably small relative to the current measurement precision required by the FOC loop. This is a validation item for the ADC test board.

Control Loop Timing Chain

Loop timing is driven by the MCPWM peripheral, not a software timer and not the ADC. The MCPWM timer generates a hardware output compare event at the PWM carrier valley — the moment of minimum switching activity and most settled phase currents. This event directly asserts the CONVST pin via GPIO output compare with hardware-grade timing precision (nanoseconds of jitter, no software involvement).

The AD7617 burst sequencer is programmed with all 8 channel pairs. A single CONVST pulse triggers the full burst. BUSY stays high for the entire burst duration. The control loop polls BUSY until it falls, then immediately executes a CPU polling SPI transaction to read all results. FOC math runs on the fresh data, then the loop waits for the next MCPWM event. 

MCPWM carrier valley (hardware, 20 kHz)
  → CONVST GPIO pulse (output compare, no CPU involvement)
    → AD7617 burst conversion (8 pairs, BUSY high throughout)
      → poll BUSY until low  ← CPU spins, nothing else to do
        → CPU polling SPI transaction (32 bytes, 40 MHz, 1-wire)
          → data in memory
            → FOC math (monitors → observer → foc_u_guy → svm_orchestrator → bridge_driver)
              → write last_frame_time
                → wait for next MCPWM event
Why CPU polling SPI, not DMA: DMA is useful when the CPU can do other work during the transfer. Here the control loop must block until data arrives — there is nothing to overlap. CPU polling transactions avoid DMA setup overhead and interrupt round-trip latency. The ESP-IDF SPI driver confirms CPU polling transactions are faster than DMA for short transfers. At 32 bytes this is the correct choice.
Why MCPWM drives timing, not the ADC: The AD7617 is externally triggered via CONVST — it converts when told, not autonomously. This inverts the timing relationship from delta-sigma ADCs (which run autonomously and signal when done). Having MCPWM drive CONVST means the loop rate is set by the PWM carrier with hardware precision, and ADC sampling is inherently synchronised to the PWM valley. A software timer ISR would add microseconds of jitter to the CONVST trigger.

Measure CONVST-to-data-ready with GPIO toggles and a logic analyser during ADC test board bring-up. The actual budget may be better or worse than this analysis.

ISR vs Task Execution — Open Decision

With the MCPWM-driven, CPU-polling architecture, the ISR vs task question has less impact than it did under the DMA-triggered model. The control loop spends most of its time either polling BUSY or doing FOC math — not sleeping in a task waiting for a notification. However the decision still matters for the small overhead at cycle boundaries.

Consideration Task (polling loop, wakes on MCPWM event) Direct ISR execution
Scheduling overhead ~5 µs task-notify latency from MCPWM ISR (forum estimate, unverified). ~10% of 50 µs frame budget. Zero. MCPWM ISR directly executes control loop code.
FreeRTOS primitives Full access — task notify, mutexes available after FOC math completes. Only FromISR variants. Acceptable if Core 1 is dedicated and cross-task communication is minimal.
Stack Per-task stack, configurable size. Observer + FOC math stack needs sizing. Shared ISR stack, global ESP-IDF config. Must be sized for full FOC math depth.
Hardware watchdog (TWDT) Task-based TWDT reset via esp_task_wdt_reset() each cycle. Must manually kick hardware timer register each cycle. Workable but bespoke.
Debuggability Standard task debugging, stack traces, RTOS-aware crash dumps. ISR debugging harder; limited RTOS context in crash reports.
Recommendation: Start with a task that polls BUSY in a tight loop after receiving an MCPWM event notification. This preserves all task-context benefits and the overhead is bounded. If the ~5 µs notification latency proves materially costly after profiling real hardware, migrating to direct ISR execution is a localised change — the FOC math itself is unaffected. Do not optimise this before measuring.

Control Loop Watchdog — Design

The watchdog detects two failure modes: (1) the AD7617 stops responding — BUSY never falls after CONVST, causing the control loop to spin indefinitely on the BUSY poll; and (2) the control loop hangs mid-execution (infinite loop, stack overflow, memory bus stall). A timeout on the BUSY poll catches case 1. A timestamp-based watchdog on Core 0 catches both.

Mechanism: foc_u_guy writes last_frame_time = xTaskGetTickCount() at the end of each successful cycle. The watchdog task on Core 0 wakes periodically and checks xTaskGetTickCount() − last_frame_time against a threshold. On timeout it kills gates directly via the LTC7066 EN GPIO, then notifies fault_responder.

Running the watchdog on Core 0 is essential — a Core 1 hang would also stall a Core 1 watchdog task, defeating the purpose. 

// Watchdog task (Core 0, PRIO 3)
void control_loop_watchdog_task(void *arg) {
    const TickType_t THRESHOLD = pdMS_TO_TICKS(WATCHDOG_TIMEOUT_MS);
    const TickType_t POLL_RATE = pdMS_TO_TICKS(WATCHDOG_POLL_MS);

    while (1) {
        vTaskDelay(POLL_RATE);

        TickType_t now  = xTaskGetTickCount();
        TickType_t last = atomic_load(&last_frame_time);

        if ((now - last) > THRESHOLD) {
            // Direct gate kill via LTC7066 EN GPIO — do not wait for fault_responder
            gpio_set_level(GATE_KILL_GPIO, 0);
            // Notify fault_responder for logging / LED indication
            xTaskNotify(fault_responder_handle,
                        FAULT_CONTROL_LOOP_TIMEOUT, eSetBits);
            // Suspend self — recovery requires system reset
            vTaskSuspend(NULL);
        }
    }
}
Gate kill ordering: The LTC7066 EN GPIO must be pulled low before notifying fault_responder. fault_responder may take several hundred µs to wake. The gate kill is hardware-speed via the EN pin — independent of firmware on Core 1.

PWM / ADC Synchronisation

Sampling at the PWM carrier valley minimises switching noise in current readings. At the valley of the center-aligned (triangle) carrier, all low-side FETs are on and switching activity has settled — this is the moment of minimum electromagnetic interference from commutation. Sampling here measures the fundamental average current rather than a switching artifact.

With MCPWM driving CONVST via output compare, synchronisation is inherent — the ADC conversion is triggered precisely at the carrier valley event. No additional alignment step is needed, provided all three phase PWM outputs are driven by the same MCPWM timer. Outputs from different timers may drift in phase relative to each other and would need explicit synchronisation.

All phases must share one MCPWM timer. The ESP32 MCPWM peripheral supports multiple operator/comparator pairs per timer — sufficient to drive all three half-bridges from a single timer, ensuring a common carrier phase for the CONVST trigger.

SPI Bus Configuration

The AD7617 requires a dedicated SPI bus. The ESP32 has two independent GP-SPI peripherals (HSPI, VSPI). The ADC takes one; the other is reserved for auxiliary peripherals (hall effect sensors for observer validation, or other sensors). This avoids bus sharing overhead between time-critical ADC transactions and lower-priority sensor reads.

Bus acquisition: The ESP-IDF spi_device_acquire_bus() / spi_device_release_bus() mechanism locks the bus for a sequence of transactions without releasing between them. Since only one device uses the ADC bus, the bus should be acquired once at startup and never released. This eliminates the ~1 µs polling_end unblocking overhead that would otherwise apply between transactions.

IO_MUX pin requirement: Maximum SPI clock speed (40 MHz) requires using IO_MUX-routed pins, not GPIO matrix-routed pins. The GPIO matrix adds ~10 ns propagation delay per signal edge which violates setup/hold margins at 40 MHz. Use the default HSPI or VSPI pin assignments on the ESP32 (SCLK, MOSI, MISO, CS must all be IO_MUX pins for that peripheral).

1-wire vs 2-wire serial mode: In 1-wire mode, SDOA and SDOB are multiplexed onto a single MISO line (32 bytes per burst read). In 2-wire mode they are separate, halving the serial clock time (16 bytes effective per burst read, saving ~3.2 µs). 1-wire is preferred to keep the pin count low and leave MISO2 available. 2-wire is the fallback if FOC math cannot fit within the ~28 µs remaining after 1-wire readout.

ISR vs task execution Start with task + MCPWM event notification. Measure actual latency with GPIO toggles on real hardware before optimising. See tradeoff table above. Architecture is compatible with either approach.
Watchdog threshold values WATCHDOG_TIMEOUT_MS (~500 µs) and WATCHDOG_POLL_MS (~250 µs) are starting points. Tune after measuring actual frame period and jitter on hardware.

AD7617 noise floor validation 14-bit SAR noise floor must be validated against actual switching noise on phase current signals. INA241 gain and shunt value must map full-scale current to sufficient ADC range. Validation item for ADC test board bring-up.
SPI transaction timing validation The 8 µs overhead figure from ESP-IDF docs is a hard upper bound but its actual value at 40 MHz with bus held acquired is unknown. Measure CONVST-to-data-ready with logic analyser during bring-up.
BUSY poll vs BUSY interrupt for SPI trigger Currently the control loop spins on BUSY. An alternative is to wire BUSY to a GPIO interrupt that triggers the SPI transaction, allowing the CPU to potentially do other work during conversion. Given conversion takes only ~7.5 µs and the CPU has nothing else to do on Core 1, spinning is probably correct — but worth confirming once the full FOC math cycle time is profiled.
